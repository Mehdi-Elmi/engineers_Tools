# الگوی مشترک پنجره‌سازی

این فایل قانون پایه همه پنجره‌های پروژه `Engineer Tools` است.

## مسیر رسمی پروژه

```text
Mehdi-Elmi/engineers_Tools
```

از مسیر قبلی `engineers_Toolses` برای توسعه جدید استفاده نشود.

## قانون اصلی

هر پنجره جدید باید ادامه همین الگو باشد، نه طراحی جداگانه از صفر. با این حال، این پروژه باید با نگاه برنامه‌نویسی و طراحی حرفه‌ای جلو برود. اگر در ادامه مشخص شد یک الگوی فعلی برای کیفیت، سرعت، تجربه کاربری یا توسعه ماژولار مناسب نیست، تغییر مسیر مجاز است؛ اما باید آگاهانه، مستند و هماهنگ با زبان بصری کل پروژه انجام شود.

## قانون زبان بصری

- طراح باید مثل یک طراح نرم‌افزار مهندسی حرفه‌ای تصمیم بگیرد، نه فقط با کپی کردن ظاهر پیش‌فرض کنترل‌ها.
- الگوهای UI قفل کور نیستند؛ اگر راه بهتر وجود داشته باشد، می‌توان طراحی را اصلاح کرد.
- تغییرات بنیادی باید در README یا همین فایل ثبت شوند تا چت‌های بعدی مسیر را اشتباه ادامه ندهند.
- دکمه‌های تأیید، `Add Page`, `Yes`, `No`, `Apply` و موارد مشابه باید از الگوی کلید حرفه‌ای مشترک استفاده کنند.
- منوهای `File`, `Edit`, `View`, `Insert`, `Draw`, `Help` باید شبیه منوی متنی نرم‌افزارهای ویندوز باشند، نه دکمه‌های برجسته.
- گزینه `Modify` در پنجره مادر حذف شده و نباید بدون دستور جدید کاربر برگردد.
- دکمه `Home` باید به صورت کلید گرافیکی واضح با آیکن خانه سه‌بعدی و رنگی باشد.
- هر چیزی که در پروژه با عنوان پنجره، دیالوگ یا منو باز می‌شود باید گوشه‌های گرد و ظاهر هماهنگ با پروژه داشته باشد.
- پنجره‌ها و dropdownها باید با mask واقعی گرد شوند تا هیچ مستطیل تیز پشت گوشه‌ها دیده نشود.

## ساختار پنجره مادر

- `TopBar`: نوار سرمه‌ای بالا با لوگو، عنوان، minimize، maximize و close.
- `CommandBar`: منوهای `Home`, `File`, `Edit`, `View`, `Insert`, `Draw`, `Help`.
- `StartBar`: نوار ابزار سریع؛ ساختار آن در فایل جدا نگهداری می‌شود.
- `Properties`: پنل تنظیمات ابزار یا آبجکت انتخاب‌شده.
- `GridCanvas`: بوم طراحی، بدون عنوان اضافه داخل Canvas.
- `Layers`: پنل لایه‌ها و ساختار صفحه.
- `PageBar`: مدیریت صفحه‌ها.
- `StatusBar`: مختصات، zoom، unit و وضعیت tool select. `Zoom` در سمت چپ نوار است و `Unit` در سمت راست باقی می‌ماند.

## قانون لوگو

لوگوی اصلی برنامه باید در پوشه زیر قرار بگیرد:

```text
logo/
```

پنجره مادر در زمان اجرا اولین فایل تصویری معتبر داخل این پوشه را به عنوان لوگوی `TopBar` می‌خواند. پس چت‌های بعدی نباید دوباره متن `AT` را به‌عنوان لوگوی اصلی طراحی کنند. اگر فایل تصویری در پوشه وجود نداشت یا قابل خواندن نبود، فقط برای fallback متن `AT` نمایش داده می‌شود.

فرمت‌های معتبر فعلی:

- `.png`
- `.jpg`
- `.jpeg`
- `.bmp`
- `.webp`

## قانون منوهای پنجره مادر

منوهای پنجره مادر در `src/engineers_tools/app/module_window.py` تعریف می‌شوند و ظاهر آن‌ها از `src/engineers_tools/app/theme.py` می‌آید. هر منو باید مثل منوی ویندوز، دقیقاً زیر همان دکمه‌ای که کاربر کلیک کرده به صورت dropdown باز شود. خود dropdown باید یک باکس گرد، شکیل و هماهنگ با پروژه باشد. منو نباید مثل دیالوگ وسط صفحه باز شود.

داخل dropdown نباید عنوان تکراری مثل `File` یا `Edit` در بالای گزینه‌ها نمایش داده شود؛ خود دکمه CommandBar نقش عنوان را دارد. عرض dropdown باید با محتوای خودش Auto Fit شود، اما همه منوها باید در محدوده جمع‌وجور و یکپارچه بمانند.

محتوای ثابت فعلی:

- `File`: `New`, `Open`, `Save`, `Save As`, `Page Setup`, `Print Setup`, `Import`, `Export`, `Properties`.
- `Edit`: `Copy`, `Cut`, `Paste`, `Move`, `Undo`, `Redo`, `Repeat Last Tools`, `Select All`, `Group`, `Ungroup`.
- `View`: `Start Bar`, `Grid`, `Ruler`, `Snap` با حالت انتخاب دایره‌ای.
- `Insert`: فعلاً فقط `Text`.
- `Draw`: فعلاً خالی است تا ابزار قلم نوری و طراحی بعداً مشخص شود.
- `Help`: dropdown ندارد؛ یک کلید است و باید صفحه Help برنامه را باز کند.

قانون انتخاب در منوها:

- گزینه‌های روشن/خاموش مثل `Grid`, `Ruler`, `Snap` باید نماد دایره‌ای داشته باشند.
- حالت فعال با دایره پر یا نقطه داخلی مشخص می‌شود.
- حالت غیرفعال با دایره خالی مشخص می‌شود.
- این الگو بعداً برای Select و سایر انتخاب‌های رادیویی/تک‌گزینه‌ای هم مبنا قرار می‌گیرد.

قانون اجرا:

- `Undo`, `Redo`, `Cut`, `Copy`, `Paste`, `Select All` اگر ویجت فعال این عملیات را داشته باشد همان عملیات واقعی را صدا می‌زنند.
- کلید `Delete` همچنان در سطح پنجره ثبت است، حتی اگر در dropdown فعلی `Edit` نمایش داده نشود.
- `Grid` واقعاً Canvas را روشن و خاموش می‌کند.
- `Start Bar` واقعاً نوار ابزار سریع را روشن و خاموش می‌کند.
- `Ruler`, `Snap`, `Repeat Last Tools`, `Group`, `Ungroup`, `Move` تا زمان ساخت موتور آبجکت‌ها وضعیت خود را در `StatusBar` اعلام می‌کنند و بعداً به منطق واقعی وصل می‌شوند.

## قانون Zoom

`StatusBar` باید یک کنترل `Zoom` داشته باشد:

- مقدار پیش‌فرض `100%` است.
- کاربر باید بتواند عدد را دستی وارد کند.
- عدد اعشاری معتبر است، مثل `100.25%`.
- فلش‌های بالا و پایین باید مشخص باشند و مقدار را با پله `5%` تغییر دهند.
- تغییر مقدار باید روی کل `GridCanvas` اعمال شود.
- `Unit` باید در سمت راست StatusBar مشخص و جدا بماند.

## قانون File Dialog

پنجره‌های `Open`, `Save`, `Save As`, `Import`, `Export` باید از پنجره اختصاصی پروژه استفاده کنند:

```text
src/engineers_tools/app/project_file_dialog.py
```

قانون‌ها:

- شروع عملیات نباید با پنجره خام `QFileDialog` باشد.
- پنجره فایل باید TopBar سرمه‌ای، نوار ناوبری تیره، `Back`, `Up`, مسیر فعلی، Places، لیست فایل‌ها، `File Name` و `File Type` داشته باشد.
- Places باید از مسیرهای واقعی سیستم مثل `Desktop`, `Downloads`, `Documents`, `Pictures`, `OneDrive` و driveها ساخته شود.
- آیکن فایل‌ها و فولدرها باید از icon provider سیستم گرفته شود تا PDF، فولدر و فایل‌های دیگر نماد مناسب خودشان را داشته باشند.
- `Open` باید PDF، فایل‌های پروژه، SVG و تصاویر را انتخاب کند و مسیر را به workspace بدهد.
- `Save` اگر فایل فعلی وجود داشته باشد همان را ذخیره می‌کند؛ اگر مسیر فعلی ندارد باید وارد `Save As` شود.
- `Save As` مسیر جدید می‌گیرد و پسوند پیش‌فرض مناسب اضافه می‌کند.
- ذخیره PDF باید فایل PDF معتبر تولید کند؛ این بخش بعداً به موتور واقعی رندر/پرینت وصل می‌شود.
- خروجی فعلی تا قبل از موتور طراحی، placeholder معتبر برای `.etools`, `.pdf`, `.svg`, `.png` می‌سازد.

## قانون New

`New` نباید پنجره قبلی را ببندد. باید یک workspace جدید از همان ماژول بسازد و پنجره قبلی همچنان در دسترس بماند. کنترل این رفتار در `src/engineers_tools/app/controller.py` انجام می‌شود.

## قانون گوشه‌ها و نوارها

- چهار گوشه پنجره باید از نظر بصری گرد دیده شوند.
- `TopBar` گوشه‌های بالایی را گرد می‌کند.
- `StatusBar` گوشه‌های پایینی را گرد می‌کند.
- پنجره مادر و dropdownها باید با mask گرد clip شوند تا هیچ پس‌زمینه مستطیلی پشت آن‌ها نماند.
- دکمه close باید علامت واقعی `×` داشته باشد، نه حرف `x`.

## قانون Minimize و Maximize

پنجره‌های ماژول frameless هستند، بنابراین maximize نباید فقط با `showMaximized()` ساده انجام شود. قانون فعلی:

- minimize باید از متد `_minimize_window()` استفاده کند.
- maximize باید دستی با `screen.availableGeometry()` انجام شود.
- هندسه قبلی پنجره باید در `_normal_geometry` ذخیره شود.
- restore باید پنجره را به همان هندسه قبلی برگرداند.
- وقتی پنجره maximize دستی است، drag با موس نباید پنجره را جابه‌جا کند.

## قانون Start Bar

`StartBar` باید مستقل از پنجره مادر نگهداری شود:

```text
src/engineers_tools/ui/start_bar.py
```

پنجره مادر فقط جایگاه Start Bar را می‌سازد و این فایل را صدا می‌زند. هر ماژول می‌تواند با override کردن متد زیر ابزارهای خودش را معرفی کند:

```text
get_start_bar_tools()
```

هر ابزار جدید باید:

- داخل `StartBarTool` تعریف شود.
- `key`, `label`, `tooltip` و در صورت نیاز `properties_schema` داشته باشد.
- دکمه مشخص در Start Bar داشته باشد.
- tooltip انگلیسی داشته باشد.
- icon style یکسان با پروژه داشته باشد.
- اگر آبجکت می‌سازد، Properties schema خودش را معرفی کند.
- اگر ابزار انتخاب است، از قانون واحد Select پیروی کند.

چت‌های اختصاصی هر پروژه نباید برای تغییر ابزارهای Start Bar ساختار پنجره مادر را خراب کنند. تغییر باید از `start_bar.py` یا override همان ماژول انجام شود.

## قانون ابزارهای ثابت مشترک

ابزارهایی مثل `Grid`, `Snap`, `Unit` و ابزارهای پایه مشابه، بعد از طراحی و تأیید، باید به‌عنوان ابزار مشترک قابل استفاده در همه برنامه‌ها نگهداری شوند. تا قبل از تأیید نهایی، فقط placeholder یا تعریف اولیه آن‌ها مجاز است.

## قانون Select

Select باید در کل پروژه یک رفتار واحد داشته باشد:

- آیکن نشانگر انتخاب واضح باشد.
- کلیک روی آبجکت باعث highlight شود.
- Properties آبجکت انتخاب‌شده نمایش داده شود.
- چند انتخابی بعداً با همان الگو توسعه داده شود.
- هیچ ماژولی حق ندارد Select جداگانه و ناسازگار بسازد.

## قانون Properties

Properties باید بر اساس ابزار فعال یا آبجکت انتخاب‌شده تغییر کند. هر ابزار باید schema تنظیمات خود را داشته باشد. نمونه‌های پایه:

- `Selection`
- `Coordinates`
- `Size`
- `Style`
- `Behavior`

## قانون `__init__.py`

`__init__.py` فایل طراحی UI نیست. این فایل فقط مرز packageهای Python را مشخص می‌کند تا importها درست کار کنند.

قانون‌ها:

- هر پوشه Python که در import استفاده می‌شود باید `__init__.py` داشته باشد.
- محتوای `__init__.py` باید خالی یا بسیار سبک باشد.
- منطق Toolbar، Properties، Canvas، Select، File Dialog و ابزارها نباید داخل `__init__.py` نوشته شود.
- چت اختصاصی Toolbar با `module_window.py` یا فایل اختصاصی Toolbar کار می‌کند، نه با `__init__.py`.
- فقط هنگام ساخت package جدید، تغییر نام مسیر import، یا export عمومی کنترل‌شده می‌توان `__init__.py` را تغییر داد.
